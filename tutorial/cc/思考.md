## 1. 背景介绍
### 1.1 程序分析“老登”做法
一般来说，我们做程序分析——顾名思义就是对程序的属性进行分析。比如：
* 整数分析：分析代码所有使用整数的地方，这个整数var的范围。
* 指针分析：分析代码所有使用指针的地方，这个指针指向那里。

最朴素的应用就是做缺陷检测——找bug：
* 整数分析：我们可以看 `a / b`的时候，`b`是否会为`0`；数组取值的时候`arr[index]`，`index`是否会超过数组的大小。
* 指针分析：我们看指针使用的时候，是否会是空指针。

那么我们怎么做嗯？
1. 我们会对程序进行建模，比如构建AST、Graph。
2. 然后：模拟执行代码，更新状态，例如：记录每个变量的情况。

这个过程会非常“累”，因为你要处理的情况非常多！而且，你要先构建AST、Graph等。

### 1.2 LLM时代
直接把代码当作文本，丢给LLM就行了。现在有无数个这样的工具。
我相信，今后大概率很少有人去走传统路线。毕竟ROI太低，而且一点不时髦。

在这个背景之下，我也在做一些AI+传统程序分析融合的探索，做一些小工具。因此，做的时候有很多“啊哈”时刻！


## 2. Agent的理解
### 2.1 Agent的本质是什么？

**所有的 Agent 本质上就是“带着特定 System Prompt 的 LLM 问答循环”。**

所以，当我习惯“流程编排”方式的逻辑的时候，一开始很不理解Agent。原因有如下几个：
1. 我习惯于把过程写死在代码里面。如果我们做一个专有的Agent那么可以理解，第一步干什么、第二步等等。但是如果泛化到普适性问题，这个是怎么做到的？
    * 理解：奥妙就在“问答循环”，Agent把原先的流程作为回答的一部分。你的主体就可以按照回答的步骤，来执行。
2. 怎么那么多“名词概念“，SubAgent、Skill、多智能体？为什么都是一个一个md文件？
    * 理解：奥妙就在“问答循环”，现在的LLM机制就是”问答“，所以我们需要转换思维，一切都是一个文本文件。
3. Agent的核心似乎就是调用LLM的API，那么Claude Code（CC）的50万行代码干啥的？
    * 理解：没错，核心代码可能就10行。差别在于各种工程细节，至少Prompt构建、结果解析、Loop机制去执行，一堆坑。
    * 所以，我认为我现在做的还不是一个Agent，只是一个AI-Powered Workflow工具。


### 2.2 我们怎么和LLM有效交互？

**结构化的输出，是工具稳定的前提。**

所以：我自己实现的代码里面，有很大一部分内容是处理AI返回的结果。这个问题早期给我带来极大的困扰。
我需要一个确定的json格式，但是AI经常会犯错。
有一些模型可以通过schema保证，比如openai。但是很多时候，一些模型总是自由发挥。
于是我有非常多的尝试和实验，最终确定一套组合方式。最近看[CC的代码](https://github.com/anthropics/claude-code-security-review/blob/main/claudecode/json_parser.py)，发现大家做法类似。

总结如下：
1. 提示词用一些强硬的语气词"You MUST output your findings as structured JSON with this exact schema:"
2. 返回结果的鲁棒性处理：正则匹配、过程匹配、json-repair
3. 重试：问题重试、结果修正


### 2.3 有效测试
**概率问题，交给概率**

我有一些代码“洁癖”，我的代码即使是AI生成，我也会一行一行的Review，甚至重写。
我的代码，每一次改动，一定会有对应的测试保证没问题。
但是，这个测试在AI模式下给予我非常大的困扰。
一方面，不同的模型行为不同，经常会有“意外”。另一方面，即使你设置temp为0，AI依旧会有不确定性。
LLM要的不就是这个不确定性嘛！所以，测试怎么办？
在我看了一些CC的代码之后，我觉得有一些想法类似！

总结如下：
1. 中间过程：结偶，构造mock输入。这个其实就是看个人意愿。
2. 流程测试：用一些确定性的输入，比如：非常简单的例子。那么你运行100次，99次成功。失败的那次流水线挂了，那再执行一次就好。
特别地，现在LLM的能力不差，所以这种情况极少发生。
3. 集成测试：这是最困扰我的地方。一开始，我有20个测试用例，判断真假。我要求100%正确。结果，很难稳定。因为不确定太容易出现。
所以我总结有效的方式如下：
    * 概率的问题，概率解决：设置置信区间，比如90%正确就好。
    * 在流水线上控制测试范围：如果改了某些核心逻辑的文件，执行批量数据测试。
    * 为了增加通过率，可以增加重试机制，如果两次分析，都没达标，那就流水线阻塞。


### 2.4 为什么要做SubAgent、多Agent编排？

**Subagent 和你直接对话用的主 Agent，在底层推理模式上毫无区别，都是“问答 + 工具调用”。唯一的不同在于它们加载的 System Prompt 不同，以及被允许使用的工具集不同。**

所以：
1. 子Agent有自己的上下文，不被污染；权限设计、失败处理等。
    * 这点非常重要！如果不拆分子Agent，那么Prompt里面的要求会非常多。那么LLM一定会“失去焦点”。
2. 专有的Agent可以独立设计：比如可以把问题做拆分、输入返回也相对独立。
    * 回顾上个问题本质是什么？—— 是一个问答循环。所以问题的输入越专有越好！所以，专有的Agent你就可以写好对应的Prompt模板！
3. 大问题拆解并行处理。
    * 这个也非常重要。当然，我们知道API访问的时候会有窗口、次数的限制。但是“分治策略”在这时候也会起作用。
    * 比如：读代码。你可以起多个Agent读，然后生成总结。
