# 协调器模式——多智能体

当单个智能体无法应对复杂工程任务时，Claude Code 提供了 Coordinator 模式——一种中心化的多智能体编排方案。

Coordinator 模式采用"协调者-工作者"架构，由一个专职协调者管理多个并行工作者的生命周期和任务分配。

## Coordinator定位
1. Coordinator 的角色定义在系统提示生成函数中，这是一段精心设计的系统提示，定义了协调者的完整行为规范。
    * 主Agent角色发生变化，不再执行而是管理。
    * worker不通信
2. 是协调者，不是执行者；复杂任务委派。
3. 四个核心工具：生成agent、停止工具、发送信息、结构化输出。
4. 工作流程：
    * Research：Workers——调研信息，例如分析文档
    * Synthesis：Coordinator——理解问题、编写实施规格，例如实施规格文档
    * Impl：Workers——按照规格进行修改，例如代码变更
    * Verification：Workers——测试修改是否正确，例如测试

## 实现
1. 激活后，为主 Agent 加载一个完全不同的系统提示词
```
export function getCoordinatorSystemPrompt(): string {
  return `You are Claude Code, an AI assistant that orchestrates 
  software engineering tasks across multiple workers.

  You are a **coordinator**. Your job is to:
  - Help the user achieve their goal
  - Direct workers to research, implement and verify code changes
  - Synthesize results and communicate with the user
  - Answer questions directly when possible — don't delegate work 
    that you can handle without tools`
}
```
2. 系统会将其工具集强制过滤为仅剩 4 个用于管理的工具
3. 在prompt里面就写好工作流程，这部分交给LLM之后，就会有对应的返回。
```
## 工作流程示例：
1. 用户: "帮我分析整个项目的安全风险"
2. 你: 拆解任务，同时启动 3 个 Worker:
   - Worker1: "分析认证模块的安全风险"
   - Worker2: "分析数据库模块的安全风险"
   - Worker3: "分析 API 接口的安全风险"
3. 你: 收集所有 Worker 的结果
4. 你: 整合并输出最终安全报告
```
4. 流程：
    * 第一轮给把上述的Prompt给LLM，返回结构化json包含具体的action和workers名字
    * worker_prompts = decision["workers"]，处理每一个worker，创建tool去执行任务。
    * 都执行结束之后，汇总信息。
